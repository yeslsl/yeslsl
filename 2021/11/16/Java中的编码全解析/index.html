  2. ISO8859-1（亦称扩展ASCII编码）

    1. iso-8859-1表示中文汉字 https://blog.csdn.net/qq_42307562/article/details/103427042
    2. ISO-8859-1编码是单字节编码，向下兼容ASCII，其编码范围是0x00-0xFF，0x00-0x7F之间完全和ASCII一致，0x80-0x9F之间是控制字符，0xA0-0xFF之间是文字符号。此字符集主要支持欧洲使用的语言

  3. GBK:

    1. 汉字国标扩展码,基本上采用了原来GB2312-80所有的汉字及码位，并涵盖了原Unicode中所有的汉字20902，总共收录了883个符号， 21003个汉字及提供了1894个造字码位。
    2. Windows 95系统就是以GBK为内码，又由于GBK同时也涵盖了Unicode所有CJK汉字，所以也可以和Unicode做一一对应。
    3. gb2312 < gbk < gb18030

  4. Unicode

    1. unicode编码目前最大到10FFFF，三个字节。 关于unicode的介绍参考 https://unicode-table.com/cn/blocks/
    2. unicode介绍：当前的Unicode字符分为17组编排，每组称为平面（Plane），而每平面拥有65536（0~FFFF）个CodePoint。当前只用了少数平面。
      1. 0号平面 U+0000 - U+FFFF 基本多文种平面（Basic Multilingual Plane，简称BMP）
      2. 1号平面 U+10000 - U+1FFFF 多文种补充平面（Supplementary Multilingual Plane，简称SMP）
      3. 2号平面 U+20000 - U+2FFFF 表意文字补充平面（Supplementary Ideographic Plane，简称SIP）
      4. 3号平面 U+30000 - U+3FFFF 表意文字第三平面（未正式使用）（Tertiary Ideographic Plane，简称TIP）
      5. 4号平面至13号平面 U+40000 - U+DFFFF（尚未使用）
      6. 14号平面 U+E0000 - U+EFFFF 特别用途补充平面（Supplementary Special-purpose Plane，简称SSP）
      7. 15号平面 U+F0000 - U+FFFFF 保留作为私人使用区（A区）（Private Use Area-A，简称PUA-A）
      8. 16号平面 U+100000 - U+10FFFF 保留作为私人使用区（B区）（Private Use Area-B，简称PUA-B）

  3. java中的代码实现

    1. Character类中，又把所有的Unicode字符分别按编码和应用角度来组织子集。

    2. UnicodeBlock是Subset的子类，代表Unicode规范中的字符，从编码角度将Unicode划分为若干子集。如：0000-007F 基本拉丁文；属于0号平面。

    3. UnicodeScript从应用角度将Unicode划分为若干子集。每一个Unicode只属于唯一一个UnicodeScript。

    4. Unicode的每一个字符都是由CodePoint表示的，每个CodePoint都是一个由Unicode标准赋予含义的唯一数字，代表一个唯一的字符。ASCII码包含128个CodePoint，范围是0到7F，扩展ASCII码包含256个CodePoint，范围是0到FF，而Unicode包含1,114,112个CodePoint，范围是0到10FFFF。Unicode码空间划分为17个Unicode字符平面（基本多文种平面，16个辅助平面），每个平面有65,536个CodePoint。因此Unicode码空间总计是17 × 65,536 = 1,114,112。

    5. 问题来了，Java的字符类型只能表示65,536个字符，那如何表示1,114,112个字符？

    6. 那要从Java的历史谈起。早期 Java 版本使用 16 位 char 数据类型表示 Unicode 字符。这种设计方法早期比较合理，因为所有 Unicode 字符拥有的值都小于 65,535 (0xFFFF)，可以通过 16 位表示。但是，Unicode 后来将最大值增加到 1,114,111 (0x10FFFF)。由于 16 位太小，不能表示 Unicode version 3.1 中的所有 Unicode 字符，因此引入了UTF-32 编码模式。但与 32 位值相比，16 位值的内存使用效率更高，因此 Unicode 引入了一种新设计方法来允许继续使用 16 位值。UTF-16 中采用的这种设计方法是在65,536个字符中分配 1,024个值给 16 位高代理（high surrogate），再分配1,024个值分配给 16 位低代理（low surrogate）。它使用一个高代理加上一个低代理组成一个代理对（surrogate pair）来表示65,536 (0x10000) 和 1,114,111 (0x10FFFF) 之间的 1,048,576个字符，即增加了1024 * 1024个字符。因此要大于65535个字符，需要引入surrogate字符。在代码中：高位代理区间：D800~DBFF， 低位代理区间：DC00~DFFF。

    7. 以上判读字符是高位代理区间，还是低位代理区间。

    8. 高位代理char+低位代理char <=> CodePoint

    9. 这样，65,536 (0x10000) 和 1,114,111 (0x10FFFF) 之间的 1,048,576个字符就可以使用两个特殊的char代替。
